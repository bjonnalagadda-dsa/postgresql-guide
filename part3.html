Lesson 01
Views
Virtual tables — save complex queries as named objects
👨‍🏫
A view is a saved SELECT query with a name. Instead of writing the same 10-line JOIN every time, you define it once as a view and then query it like a table. The view doesn't store any data — every time you SELECT from it, it runs the underlying query. Think of it as a named window into your data.
sql — Creating and using views
-- Without a view, every dashboard query needs this JOIN:
SELECT
    o.id, u.username, u.email,
    o.total, o.status, o.created_at
FROM orders o
JOIN users u ON u.id = o.user_id;

-- Create a view — save it once, use it everywhere
CREATE VIEW orders_with_users AS
SELECT
    o.id          AS order_id,
    u.id          AS user_id,
    u.username,
    u.email,
    o.total,
    o.status,
    o.created_at,
    o.updated_at
FROM orders o
JOIN users u ON u.id = o.user_id;

-- Now query it like a table
SELECT * FROM orders_with_users
WHERE status = 'pending';

SELECT username, SUM(total) AS lifetime_value
FROM orders_with_users
GROUP BY username
ORDER BY lifetime_value DESC;

More view patterns

sql — Real-world view examples
-- A "safe" view that hides sensitive columns
CREATE VIEW public_users AS
SELECT id, username, bio, avatar_url, created_at
FROM users
WHERE status = 'active';
-- Now grant public_users to API role — no password_hash ever exposed

-- Aggregated stats view
CREATE VIEW product_sales_summary AS
SELECT
    p.id,
    p.name,
    p.price,
    COUNT(oi.id)         AS times_ordered,
    SUM(oi.quantity)     AS total_units_sold,
    SUM(oi.quantity * oi.unit_price) AS total_revenue
FROM products p
LEFT JOIN order_items oi ON oi.product_id = p.id
GROUP BY p.id, p.name, p.price;

-- Update or replace a view
CREATE OR REPLACE VIEW public_users AS
SELECT id, username, bio, avatar_url, created_at, is_verified
FROM users
WHERE status = 'active';

-- Drop a view
DROP VIEW IF EXISTS public_users;

-- List all views in the current database
SELECT table_name FROM information_schema.views
WHERE table_schema = 'public';

Updatable Views

Simple views (no JOIN, no GROUP BY, no DISTINCT, no aggregate) are automatically updatable — you can INSERT/UPDATE/DELETE through them. Complex views need a RULE or INSTEAD OF trigger.

sql — Updatable view
CREATE VIEW active_users AS
SELECT * FROM users WHERE status = 'active';

-- This works — updates the underlying users table
UPDATE active_users
SET is_verified = TRUE
WHERE id = 5;

-- WITH CHECK OPTION prevents rows from "escaping" the view's WHERE
CREATE VIEW active_users AS
SELECT * FROM users WHERE status = 'active'
WITH CHECK OPTION;
-- Now you can't INSERT/UPDATE a row to status='suspended' through this view
💡 Best uses for views
  • Security: expose only the columns a role needs — hide passwords, PII
  • Simplicity: give complex JOINs a clean name for your API layer
  • Backwards compatibility: rename a column in the table but keep the old name in a view
Lesson 02
Materialized Views
Pre-computed result sets — trading freshness for speed
👨‍🏫
A materialized view stores the query result on disk — like a snapshot. Reading it is instant because no computation happens at query time. The tradeoff: the data can be stale until you REFRESH it. Use them for expensive aggregation queries that power dashboards, reports, or analytics where a few minutes of delay is acceptable.
sql — Materialized views
-- This query is expensive — scans all orders, all items, all products
-- Running it on every dashboard load is wasteful
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT
    DATE_TRUNC('day', o.created_at) AS day,
    COUNT(DISTINCT o.id)             AS order_count,
    COUNT(DISTINCT o.user_id)        AS unique_customers,
    SUM(o.total)                     AS revenue,
    AVG(o.total)                     AS avg_order_value
FROM orders o
WHERE o.status = 'paid'
GROUP BY DATE_TRUNC('day', o.created_at)
ORDER BY day;

-- Query the materialized view — always instant
SELECT * FROM daily_revenue
WHERE day >= NOW() - INTERVAL '30 days';

-- You can index materialized views just like tables!
CREATE INDEX idx_daily_revenue_day ON daily_revenue(day);

-- Refresh: re-run the underlying query and update the stored data
REFRESH MATERIALIZED VIEW daily_revenue;

-- Refresh WITHOUT locking reads (PostgreSQL 9.4+)
-- Requires a UNIQUE index on the view
CREATE UNIQUE INDEX idx_daily_revenue_day_uniq ON daily_revenue(day);
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;

-- Drop it
DROP MATERIALIZED VIEW daily_revenue;
ViewMaterialized View
Data storageNone — runs query each timeStored on disk like a table
SpeedDepends on underlying queryAlways fast (just a table read)
FreshnessAlways up to dateStale until REFRESH
IndexesNot directlyYes, like any table
Best forSimple queries, security layersExpensive aggregations, dashboards
📌 Auto-refresh pattern
Schedule REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue; via pg_cron (a PostgreSQL extension), a cron job, or your app's background job scheduler. Run it every 15 minutes for near-real-time dashboards.
Lesson 03
Functions & PL/pgSQL
Write reusable logic that lives inside the database
👨‍🏫
PostgreSQL has its own programming language built in — PL/pgSQL. It looks like Pascal mixed with SQL. You can write functions with variables, IF/ELSE, loops, and error handling — all running inside the database engine. This is useful for complex calculations, data validation, and any logic you want to centralize rather than repeat in every app.

Simple SQL functions (no PL/pgSQL needed)

sql — Pure SQL functions
-- A simple utility function
CREATE OR REPLACE FUNCTION full_name(
    first_name TEXT,
    last_name  TEXT
) RETURNS TEXT
LANGUAGE SQL IMMUTABLE  -- always same output for same input (cacheable)
AS $$
    SELECT TRIM(first_name || ' ' || last_name);
$$;

-- Use it in a query
SELECT full_name(first_name, last_name) FROM users;

-- A function that queries a table
CREATE OR REPLACE FUNCTION user_order_count(p_user_id BIGINT)
RETURNS INT
LANGUAGE SQL STABLE  -- reads DB, doesn't modify it
AS $$
    SELECT COUNT(*)::INT FROM orders WHERE user_id = p_user_id;
$$;

-- Call it
SELECT username, user_order_count(id) AS orders
FROM users;

PL/pgSQL — full programming language

plpgsql — Variables, IF, loops
-- Function with variables, IF/ELSE, exception handling
CREATE OR REPLACE FUNCTION apply_discount(
    p_order_id  BIGINT,
    p_discount  NUMERIC  -- percentage, e.g. 10 means 10%
) RETURNS NUMERIC
LANGUAGE plpgsql
AS $$
DECLARE
    v_current_total  NUMERIC;
    v_new_total      NUMERIC;
BEGIN
    -- Get the current order total
    SELECT total INTO v_current_total
    FROM orders
    WHERE id = p_order_id;

    -- Check the order exists
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Order % not found', p_order_id;
    END IF;

    -- Validate discount range
    IF p_discount < 0 OR p_discount > 100 THEN
        RAISE EXCEPTION 'Discount must be between 0 and 100';
    END IF;

    -- Calculate and update
    v_new_total := v_current_total * (1 - p_discount / 100.0);

    UPDATE orders
    SET total = v_new_total, updated_at = NOW()
    WHERE id = p_order_id;

    RETURN v_new_total;
END;
$$;

-- Call it
SELECT apply_discount(42, 15);  -- 15% off order #42
plpgsql — Loops and RETURN TABLE
-- Function that returns a TABLE (set of rows)
CREATE OR REPLACE FUNCTION top_customers(p_limit INT DEFAULT 10)
RETURNS TABLE(
    username       TEXT,
    email          TEXT,
    total_spent    NUMERIC,
    order_count    BIGINT
)
LANGUAGE plpgsql STABLE
AS $$
BEGIN
    RETURN QUERY
        SELECT
            u.username::TEXT,
            u.email::TEXT,
            SUM(o.total),
            COUNT(o.id)
        FROM users u
        JOIN orders o ON o.user_id = u.id
        WHERE o.status = 'paid'
        GROUP BY u.id, u.username, u.email
        ORDER BY SUM(o.total) DESC
        LIMIT p_limit;
END;
$$;

-- Call it like a table
SELECT * FROM top_customers(5);
SELECT * FROM top_customers();  -- uses default 10
VolatilityMeaningCan be optimized?
IMMUTABLESame inputs always produce same output. No DB access.✅ Result can be cached forever
STABLESame inputs return same result within one query. Reads DB.✅ Can use indexes, cached per query
VOLATILEMay return different results each call (default). Can modify DB.❌ Never cached
💡 Always declare volatility
Adding IMMUTABLE or STABLE when appropriate lets the planner optimize your function calls significantly. A function used in a WHERE clause with the correct volatility can be pushed down and use indexes.
Lesson 04
Stored Procedures
Functions that can manage their own transactions
👨‍🏫
Stored procedures look almost identical to functions but have one superpower: they can COMMIT and ROLLBACK transactions themselves. A function always runs inside the caller's transaction. A procedure can start, commit, or roll back sub-transactions independently. Use procedures for long batch operations where you want to commit progress in chunks.
plpgsql — Stored procedure (CALL, not SELECT)
-- Procedure: process all pending orders
CREATE OR REPLACE PROCEDURE process_pending_orders()
LANGUAGE plpgsql
AS $$
DECLARE
    v_order    orders%ROWTYPE;
    v_count    INT := 0;
BEGIN
    FOR v_order IN
        SELECT * FROM orders
        WHERE status = 'pending'
        ORDER BY created_at
        LIMIT 1000  -- process in batches
    LOOP
        -- Mark as processing
        UPDATE orders
        SET status = 'processing', updated_at = NOW()
        WHERE id = v_order.id;

        v_count := v_count + 1;

        -- Commit every 100 rows so we don't hold one huge transaction
        IF v_count % 100 = 0 THEN
            COMMIT;  -- ← only procedures can do this, not functions
        END IF;
    END LOOP;

    COMMIT;  -- final commit
    RAISE NOTICE 'Processed % orders', v_count;
END;
$$;

-- Call a procedure with CALL (not SELECT)
CALL process_pending_orders();
FunctionProcedure
Returns valueYes — RETURNS typeNo (use OUT parameters)
Can COMMIT/ROLLBACK❌ No✅ Yes
Called withSELECT fn() or in expressionsCALL proc()
Use forCalculations, returning dataBatch jobs, ETL, multi-step DML
Lesson 05
Triggers
Auto-execute code on INSERT, UPDATE, or DELETE
👨‍🏫
A trigger is a function that fires automatically when something happens to a table. You define WHEN it fires (BEFORE or AFTER), WHAT operation (INSERT, UPDATE, DELETE), and WHICH table. PostgreSQL calls your trigger function every time. The two most common uses: auto-updating updated_at timestamps, and keeping an audit log of every change.
plpgsql — Auto-update updated_at (every app needs this)
-- Step 1: create the trigger FUNCTION
-- Trigger functions return TRIGGER type and take no args
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    NEW.updated_at := NOW();  -- NEW = the row being inserted/updated
    RETURN NEW;
END;
$$;

-- Step 2: attach the trigger to a table
CREATE TRIGGER trg_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();

-- Add to orders table too (reuse the same function!)
CREATE TRIGGER trg_orders_updated_at
    BEFORE UPDATE ON orders
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();

-- Now every UPDATE automatically sets updated_at = NOW()
UPDATE users SET email = 'new@example.com' WHERE id = 1;
-- updated_at is now current timestamp — no need to set it manually
plpgsql — Audit log trigger (track every change)
-- Create an audit table
CREATE TABLE audit_log (
    id         BIGSERIAL PRIMARY KEY,
    table_name TEXT NOT NULL,
    operation  TEXT NOT NULL,       -- INSERT / UPDATE / DELETE
    row_id     BIGINT,
    old_data   JSONB,               -- what it was before
    new_data   JSONB,               -- what it became after
    changed_by TEXT DEFAULT current_user,
    changed_at TIMESTAMPTZ DEFAULT NOW()
);

-- Generic audit trigger function (reusable on any table)
CREATE OR REPLACE FUNCTION audit_trigger_fn()
RETURNS TRIGGER
LANGUAGE plpgsql SECURITY DEFINER
AS $$
BEGIN
    IF (TG_OP = 'INSERT') THEN
        INSERT INTO audit_log (table_name, operation, row_id, new_data)
        VALUES (TG_TABLE_NAME, 'INSERT', NEW.id, to_jsonb(NEW));

    ELSIF (TG_OP = 'UPDATE') THEN
        INSERT INTO audit_log (table_name, operation, row_id, old_data, new_data)
        VALUES (TG_TABLE_NAME, 'UPDATE', NEW.id, to_jsonb(OLD), to_jsonb(NEW));

    ELSIF (TG_OP = 'DELETE') THEN
        INSERT INTO audit_log (table_name, operation, row_id, old_data)
        VALUES (TG_TABLE_NAME, 'DELETE', OLD.id, to_jsonb(OLD));
    END IF;

    RETURN NULL;  -- AFTER triggers on row: return value is ignored
END;
$$;

-- Attach to users table
CREATE TRIGGER trg_users_audit
    AFTER INSERT OR UPDATE OR DELETE ON users
    FOR EACH ROW
    EXECUTE FUNCTION audit_trigger_fn();

-- Now every change is recorded automatically
UPDATE users SET status = 'suspended' WHERE id = 3;
SELECT * FROM audit_log ORDER BY changed_at DESC LIMIT 5;

Trigger timing and scope

OptionMeaningUse case
BEFOREFires before the operation. Can modify NEW to change what gets written.set_updated_at, data normalisation, validation
AFTERFires after the operation. Row is already written.Audit logs, notifying other systems
INSTEAD OFReplaces the operation entirely. Only on views.Making complex views updatable
FOR EACH ROWFires once per affected row.Most triggers
FOR EACH STATEMENTFires once per SQL statement, even if 0 or 1000 rows affected.Logging that a statement ran

Trigger special variables

plpgsql — Trigger variables reference
NEW          -- The new row (INSERT, UPDATE). NULL for DELETE.
OLD          -- The old row (UPDATE, DELETE). NULL for INSERT.
TG_OP        -- 'INSERT', 'UPDATE', or 'DELETE'
TG_TABLE_NAME -- Name of the table that fired the trigger
TG_WHEN      -- 'BEFORE' or 'AFTER'
TG_NARGS     -- Number of args passed to trigger
TG_ARGV      -- Array of args: TG_ARGV[0], TG_ARGV[1]...

-- Example: trigger that only runs when a specific column changes
CREATE OR REPLACE FUNCTION on_status_change()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF OLD.status IS DISTINCT FROM NEW.status THEN
        RAISE NOTICE 'Order % changed from % to %',
            NEW.id, OLD.status, NEW.status;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_order_status
    BEFORE UPDATE OF status ON orders  -- only fires when status column changes
    FOR EACH ROW EXECUTE FUNCTION on_status_change();
🧠 You want a trigger that fires once when you run DELETE FROM orders WHERE status = 'cancelled' regardless of how many rows it deletes. Which trigger type do you use?
✅ B is correct. FOR EACH STATEMENT fires once per SQL statement regardless of how many rows were affected. FOR EACH ROW fires once for every affected row — on a DELETE of 10,000 rows that's 10,000 trigger calls. Use STATEMENT-level triggers when you care that the operation happened, not about individual rows.
Lesson 06
Transactions in Depth
Isolation levels, savepoints, deadlocks, advisory locks
👨‍🏫
You learned BEGIN/COMMIT/ROLLBACK in Part 1. Now let's go deeper. Real apps run hundreds of concurrent transactions. Isolation levels control what one transaction can see while another is still running. Get this wrong and you get dirty reads, phantom rows, or lost updates. PostgreSQL defaults are excellent — but you need to understand them to use them right.

Isolation levels

LevelDirty ReadNon-Repeatable ReadPhantom ReadUse when
READ UNCOMMITTEDPossible*YesYesNever in Postgres (treated as READ COMMITTED)
READ COMMITTEDNoYesYesDefault. Good for most OLTP work.
REPEATABLE READNoNoNo*Reports that must see a consistent snapshot across multiple queries
SERIALIZABLENoNoNoFinancial operations, anything requiring strict correctness
sql — Setting isolation levels
-- Default: READ COMMITTED (99% of the time, this is correct)
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- REPEATABLE READ: every SELECT in this transaction sees the same snapshot
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT SUM(balance) FROM accounts;  -- sees snapshot at BEGIN time
-- ... do more reads ...
SELECT SUM(balance) FROM accounts;  -- same result even if others committed changes
COMMIT;

-- SERIALIZABLE: maximum safety, may retry on conflict
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- ... your queries ...
COMMIT;  -- may raise "ERROR: could not serialize access" — your app must retry

Savepoints — partial rollbacks

sql — Savepoints
BEGIN;
    INSERT INTO orders (user_id, total) VALUES (1, 99.99);

    SAVEPOINT after_order;  -- mark a point we can roll back to

    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (1, 999, 2, 49.99);  -- product 999 doesn't exist — this will fail

    -- If item insert failed, roll back ONLY to the savepoint
    ROLLBACK TO SAVEPOINT after_order;

    -- The order INSERT is still in effect — we can continue
    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (1, 1, 2, 49.99);  -- product 1 exists — this works

    RELEASE SAVEPOINT after_order;  -- clean up savepoint
COMMIT;

Deadlocks and how to avoid them

sql — Deadlock example and prevention
-- DEADLOCK: Transaction A and B lock rows in opposite order
-- Txn A: locks row 1, then tries to lock row 2
-- Txn B: locks row 2, then tries to lock row 1 → deadlock!

-- PREVENTION: Always lock rows in the same order
-- Use ORDER BY when doing multi-row updates:

-- Bad: may deadlock if two transactions do this simultaneously
UPDATE accounts SET balance = balance - 100 WHERE id = p_from;
UPDATE accounts SET balance = balance + 100 WHERE id = p_to;

-- Good: lock the smaller ID first, always consistent order
SELECT * FROM accounts
WHERE id IN (p_from, p_to)
ORDER BY id
FOR UPDATE;  -- explicitly lock both rows in a consistent order

-- FOR UPDATE: lock rows so no one else can update until you commit
SELECT * FROM products
WHERE id = 5
FOR UPDATE;

-- FOR UPDATE SKIP LOCKED: skip rows already locked (queue worker pattern)
SELECT * FROM orders
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;  -- safe for multiple workers processing a queue
📌 FOR UPDATE SKIP LOCKED
This is the right way to build a job queue in PostgreSQL. Multiple workers can each grab a different row without blocking each other. This pattern replaces Redis queues for many use cases.
Lesson 07
JSONB Deep Dive
PostgreSQL's killer feature — SQL + document database in one
👨‍🏫
JSONB is the reason many teams choose PostgreSQL over MySQL. It stores JSON as binary (parsed and indexed), so you can query inside JSON fields with full index support — something MySQL can't match. You get the flexibility of a document database combined with the power of relational joins. This is why many teams skip MongoDB entirely.

JSON vs JSONB — always use JSONB

JSONJSONB
StorageRaw text — preserves whitespace, key order, duplicatesBinary parsed — normalised, no whitespace, sorted keys
Input speedFaster (no parsing)Slightly slower (parses on write)
Query speedSlower (re-parses every time)Much faster (already parsed)
IndexableNo✅ Yes — GIN index
UseAlmost neverAlways
sql — JSONB operators and path queries
-- Sample: users table with JSONB preferences column
-- {"theme":"dark","language":"en","notifications":true,"tags":["admin","beta"]}

── READ OPERATORS ────────────────────────────────────

-- -> returns JSONB (keep as JSON object)
SELECT preferences->'theme' FROM users;
-- Result: "dark"  (with quotes — it's JSONB type)

-- ->> returns TEXT (cast to plain string)
SELECT preferences->>'theme' FROM users;
-- Result: dark  (no quotes — use this in WHERE clauses)

-- Nested path using #> and #>>
SELECT preferences#>>'{address,city}' FROM users;

-- Filter by JSONB value
SELECT id, username FROM users
WHERE preferences->>'theme' = 'dark';

-- @> containment: does this JSONB contain this sub-document?
SELECT * FROM users
WHERE preferences @> '{"theme":"dark","notifications":true}'::jsonb;

-- ? key exists operator
SELECT * FROM users
WHERE preferences ? 'theme';

-- ?| any of these keys exist
SELECT * FROM users
WHERE preferences ?| ARRAY['theme', 'language'];

-- ?& all of these keys exist
SELECT * FROM users
WHERE preferences ?& ARRAY['theme', 'notifications'];

── WRITE OPERATORS ───────────────────────────────────

-- || merge/update a JSONB field
UPDATE users
SET preferences = preferences || '{"theme":"light"}'::jsonb
WHERE id = 1;

-- - delete a key
UPDATE users
SET preferences = preferences - 'language'
WHERE id = 1;

-- jsonb_set: update a nested key
UPDATE users
SET preferences = jsonb_set(preferences, '{address,city}', '"New York"')
WHERE id = 1;
sql — JSONB functions
-- Expand JSONB object to key-value rows
SELECT key, value
FROM users, jsonb_each(preferences)
WHERE id = 1;

-- Extract all keys
SELECT jsonb_object_keys(preferences) FROM users WHERE id = 1;

-- Build JSONB from columns
SELECT jsonb_build_object('id', id, 'name', username, 'email', email)
FROM users LIMIT 3;

-- Aggregate rows into a JSON array
SELECT jsonb_agg(jsonb_build_object('id', id, 'name', username))
FROM users;

-- Index for fast JSONB queries (essential for large tables)
CREATE INDEX idx_users_prefs_gin ON users USING GIN(preferences);

-- Index for a specific JSONB path (smaller, faster than full GIN)
CREATE INDEX idx_users_theme ON users((preferences->>'theme'));
⚠️ Don't over-use JSONB
JSONB is powerful but it's not a replacement for proper relational columns. Rule of thumb: if you query a field in WHERE, JOIN, or ORDER BY regularly — make it a real column. Use JSONB for truly flexible metadata where the shape varies per row (user preferences, product attributes, event payloads).
Lesson 08
Full-Text Search
Built-in search engine — no Elasticsearch needed for most apps
👨‍🏫
PostgreSQL has a full-text search engine built in — and it's surprisingly powerful. It handles stemming (run/running/ran all match "run"), stop words (ignore "the", "a"), ranking by relevance, and phrase search. For most apps with a search box, you don't need Elasticsearch. You need tsvector and a GIN index.

The core concepts

ConceptWhat it isExample
tsvectorA preprocessed, searchable document. Words are stemmed, stop words removed.'fast:1 run:2 user:3'
tsqueryA search query with AND (&), OR (|), NOT (!), phrase (<->).'fast & run'
@@The match operator — does this tsvector match this tsquery?doc @@ query
ts_rankRelevance score — how well does the document match the query?Higher = more relevant
sql — Full-text search step by step
-- Step 1: understand tsvector
SELECT to_tsvector('english', 'Fast running users are searching for products');
-- 'fast':1 'product':7 'run':2 'search':6 'user':3
-- Notice: 'are','for' removed (stop words), 'running'→'run' (stemmed)

-- Step 2: understand tsquery
SELECT to_tsquery('english', 'running & search');
-- 'run' & 'search'  ← stemmed automatically

-- Step 3: the @@ match operator
SELECT
    to_tsvector('english', 'Fast running users')
    @@
    to_tsquery('english', 'run');
-- true  (because 'running' stems to 'run')

-- Step 4: search products
SELECT name, description
FROM products
WHERE
    to_tsvector('english', name || ' ' || COALESCE(description,''))
    @@
    plainto_tsquery('english', 'wireless headphones');
-- plainto_tsquery converts plain text query to tsquery automatically

-- Step 5: rank results by relevance
SELECT
    name,
    ts_rank(
        to_tsvector('english', name || ' ' || COALESCE(description,'')),
        plainto_tsquery('english', 'wireless headphones')
    ) AS relevance
FROM products
WHERE
    to_tsvector('english', name || ' ' || COALESCE(description,''))
    @@
    plainto_tsquery('english', 'wireless headphones')
ORDER BY relevance DESC;

The right way — generated column + GIN index

Calling to_tsvector() on every query is expensive. The correct approach is to pre-compute the tsvector once and store it in a generated column, then index it.

sql — Production-ready full-text search
-- Add a generated tsvector column to products
ALTER TABLE products
ADD COLUMN search_vector TSVECTOR
    GENERATED ALWAYS AS (
        to_tsvector('english',
            COALESCE(name, '') || ' ' ||
            COALESCE(description, '') || ' ' ||
            COALESCE(category, '')
        )
    ) STORED;

-- Index it
CREATE INDEX idx_products_fts ON products USING GIN(search_vector);

-- Now searching is fast and clean
SELECT
    name, category, price,
    ts_rank(search_vector, query) AS rank
FROM products,
     plainto_tsquery('english', 'wireless headphone') AS query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

-- Phrase search: "noise cancelling" (words must appear in this order)
SELECT name FROM products
WHERE search_vector @@ phraseto_tsquery('english', 'noise cancelling');

-- websearch_to_tsquery: lets users type Google-style queries
-- "wireless headphones" -sony   means: has both words, not sony
SELECT name FROM products
WHERE search_vector @@ websearch_to_tsquery('english', '"wireless headphones" -sony');

-- Highlight matching words in results (for UI snippets)
SELECT
    name,
    ts_headline('english', description,
        plainto_tsquery('english', 'wireless headphones'),
        'MaxWords=20,MinWords=5,StartSel=<mark>,StopSel=</mark>'
    ) AS snippet
FROM products
WHERE search_vector @@ plainto_tsquery('english', 'wireless headphones');
💡 When to use PostgreSQL FTS vs Elasticsearch
Use PostgreSQL FTS when: your search is over data already in Postgres, you need relevance ranking + simple search, and you have <10M documents.
Use Elasticsearch when: you need complex faceting, fuzzy matching, autocomplete, synonyms, or are indexing terabytes of text across multiple sources.

  • Materialized views — store pre-computed results on disk; REFRESH CONCURRENTLY without locking
  • Index materialized views just like tables for instant query performance
  • SQL functions (IMMUTABLE/STABLE/VOLATILE) and RETURNS TABLE pattern
  • PL/pgSQL — DECLARE, BEGIN/END, IF/ELSIF/ELSE, FOR loops, RAISE EXCEPTION/NOTICE
  • Stored procedures — can COMMIT/ROLLBACK inside; call with CALL not SELECT
  • Triggers — BEFORE/AFTER, FOR EACH ROW/STATEMENT; NEW, OLD, TG_OP variables
  • Auto-updating updated_at and audit logs — the two most important trigger patterns
  • Transaction isolation levels — READ COMMITTED (default) vs REPEATABLE READ vs SERIALIZABLE
  • Savepoints for partial rollbacks within a transaction
  • FOR UPDATE SKIP LOCKED — the right way to build a job queue in PostgreSQL
  • JSONB operators: ->, ->>, @>, ?, ||, jsonb_set, jsonb_agg, GIN indexing
  • Full-text search: tsvector, tsquery, @@, ts_rank, ts_headline, generated columns + GIN index
