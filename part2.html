Lesson 01
Foreign Keys
Linking tables together safely
👨‍🏫
A foreign key is a contract between two tables. It says: "every value in this column MUST exist as a primary key in that other table." Without foreign keys, you can have orders that belong to user ID 999 even though user 999 was deleted. PostgreSQL enforces this for you automatically.

Let's build the classic e-commerce schema: users → orders → order_items → products. This is the foundation you'll see in almost every real app.

sql — Build the schema from scratch
-- (users table already exists from Part 1)

-- Products table
CREATE TABLE products (
    id          BIGSERIAL PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    description TEXT,
    price       NUMERIC(10,2) NOT NULL CHECK (price >= 0),
    stock_qty   INT NOT NULL DEFAULT 0 CHECK (stock_qty >= 0),
    category    VARCHAR(100),
    is_active   BOOLEAN NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Orders table — references users
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    status      VARCHAR(20) NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending','paid','shipped','delivered','cancelled')),
    total       NUMERIC(12,2) NOT NULL DEFAULT 0,
    notes       TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- The foreign key: user_id must exist in users.id
    CONSTRAINT fk_orders_user
        FOREIGN KEY (user_id)
        REFERENCES users(id)
        ON DELETE RESTRICT    -- prevent deleting a user who has orders
        ON UPDATE CASCADE     -- if user id changes, update here too
);

-- Order items — references both orders and products
CREATE TABLE order_items (
    id          BIGSERIAL PRIMARY KEY,
    order_id    BIGINT NOT NULL,
    product_id  BIGINT NOT NULL,
    quantity    INT NOT NULL CHECK (quantity > 0),
    unit_price  NUMERIC(10,2) NOT NULL,  -- snapshot price at purchase time

    FOREIGN KEY (order_id)   REFERENCES orders(id)   ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
);

ON DELETE / ON UPDATE actions

ActionWhat happens when parent row is deleted/updatedUse when
RESTRICTError — block the delete/update if children existUsers with orders, products with purchases
CASCADEAutomatically delete/update all child rows tooOrder → order_items (items mean nothing without order)
SET NULLSet the foreign key column to NULL in child rowsComments where the author deleted their account
SET DEFAULTSet to the column's DEFAULT valueRare — assign to a "deleted user" placeholder
NO ACTIONLike RESTRICT but checked at end of transactionDefault if you omit the clause
⚠️ Common mistake
Never store a price from products.price in order_items. Product prices change over time. Always snapshot the price at purchase time into order_items.unit_price. This is why that column exists.

Adding foreign keys to existing tables

sql — ALTER TABLE ADD CONSTRAINT
-- Add a foreign key after the fact
ALTER TABLE orders
ADD CONSTRAINT fk_orders_user
FOREIGN KEY (user_id) REFERENCES users(id);

-- Drop a foreign key constraint
ALTER TABLE orders
DROP CONSTRAINT fk_orders_user;

-- List all foreign keys in the database
SELECT
    tc.table_name, kcu.column_name,
    ccu.table_name AS foreign_table,
    ccu.column_name AS foreign_column
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
    ON ccu.constraint_name = tc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY';
Lesson 02
Relationship Patterns
One-to-many, many-to-many, one-to-one
👨‍🏫
Every relationship between tables fits into one of three patterns. Understanding which pattern you need tells you exactly how to structure your tables. Get this wrong and queries become painful. Get it right and everything falls into place.

1️⃣ One-to-Many

One user has many orders. One order has many items. Most common pattern. Put a foreign key on the "many" side pointing to the "one" side.

🔀 Many-to-Many

One product has many tags. One tag belongs to many products. Requires a junction table in between (e.g. product_tags).

1️⃣ One-to-One

One user has one profile. Either add a UNIQUE foreign key or merge the tables. Use when separating optional or large data.

Many-to-Many: Products ↔ Tags

sql — Junction table pattern
CREATE TABLE tags (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE
);

-- The junction table — just two foreign keys
CREATE TABLE product_tags (
    product_id BIGINT NOT NULL REFERENCES products(id) ON DELETE CASCADE,
    tag_id     INT    NOT NULL REFERENCES tags(id)     ON DELETE CASCADE,
    PRIMARY KEY (product_id, tag_id)  -- composite PK prevents duplicates
);

-- Insert data
INSERT INTO tags (name) VALUES ('electronics'),('sale'),('new'),('bestseller');
INSERT INTO product_tags (product_id, tag_id) VALUES (1,1),(1,3),(2,1),(2,2);

-- Query: get all tags for product 1
SELECT t.name
FROM tags t
JOIN product_tags pt ON pt.tag_id = t.id
WHERE pt.product_id = 1;

-- Query: get all products for tag 'sale'
SELECT p.name, p.price
FROM products p
JOIN product_tags pt ON pt.product_id = p.id
JOIN tags t          ON t.id = pt.tag_id
WHERE t.name = 'sale';

One-to-One: User ↔ Profile

sql
CREATE TABLE user_profiles (
    user_id      BIGINT PRIMARY KEY  -- PK is also the FK (enforces 1:1)
                 REFERENCES users(id) ON DELETE CASCADE,
    website      TEXT,
    twitter      VARCHAR(100),
    github       VARCHAR(100),
    location     VARCHAR(200),
    full_bio     TEXT   -- large optional data — good reason to split
);
Lesson 03
JOINs Deep Dive
INNER, LEFT, RIGHT, FULL, CROSS, SELF — when and how to use each
👨‍🏫
JOINs are how you answer questions that span multiple tables. "What did each user order?" — that's a JOIN. "Which users have never placed an order?" — that's a LEFT JOIN with a NULL check. Mastering JOINs means mastering 80% of real-world SQL.
A
B

INNER JOIN

Only rows where both sides match. Most common. Use when you only want rows that have a related record.

A
B

LEFT JOIN

All rows from left table, matching rows from right. NULL on right where no match. Use when the right side is optional.

A
B

RIGHT JOIN

All rows from right table. Rarely used — you can always rewrite as LEFT JOIN by swapping table order.

A
B

FULL OUTER JOIN

All rows from both sides. NULL where no match on either side. Use for reconciliation queries.

A × B

CROSS JOIN

Every row in A paired with every row in B. 10 rows × 10 rows = 100 rows. Use for generating combinations.

A ↺ A

SELF JOIN

Join a table to itself. Classic use: employees and their managers, both in the same table.

sql — INNER JOIN: orders with user info
-- Get all orders with the username of who placed them
-- INNER JOIN = only orders where user_id matches a real user
SELECT
    o.id          AS order_id,
    u.username,
    u.email,
    o.status,
    o.total,
    o.created_at
FROM orders o
INNER JOIN users u ON u.id = o.user_id
ORDER BY o.created_at DESC;
sql — LEFT JOIN: all users, even those with no orders
-- All users + their order count. Users with 0 orders still appear.
SELECT
    u.id,
    u.username,
    COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.username
ORDER BY order_count DESC;

-- Find users who have NEVER placed an order
SELECT u.id, u.username, u.email
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.id IS NULL;  -- ← the classic "anti-join" pattern
sql — Multi-table JOIN: order with items and product names
-- Full order detail — joining 4 tables
SELECT
    o.id          AS order_id,
    u.username,
    p.name        AS product_name,
    oi.quantity,
    oi.unit_price,
    (oi.quantity * oi.unit_price) AS line_total,
    o.status
FROM orders o
JOIN users       u  ON u.id  = o.user_id
JOIN order_items oi ON oi.order_id  = o.id
JOIN products    p  ON p.id  = oi.product_id
WHERE o.id = 1
ORDER BY p.name;
sql — SELF JOIN: employees and their managers
-- Imagine employees table has a manager_id column
ALTER TABLE users ADD COLUMN manager_id BIGINT REFERENCES users(id);

-- Self-join: employee + their manager's name
SELECT
    e.username  AS employee,
    m.username  AS manager
FROM users e
LEFT JOIN users m ON m.id = e.manager_id
ORDER BY m.username, e.username;
💡 Rule of thumb
Use INNER JOIN when the relationship must exist (orders must have a user). Use LEFT JOIN when the right side is optional (users may or may not have orders). You'll use LEFT JOIN far more often than you expect.
🧠 You want all products, and for each one show the number of times it was ordered (including products with zero orders). Which JOIN do you use?
✅ B is correct. LEFT JOIN from products keeps all products. COUNT(oi.id) counts only non-NULL order_item rows — products never ordered get count 0. An INNER JOIN would drop products with zero orders entirely.
Lesson 04
Aggregates & GROUP BY
Summarising data — COUNT, SUM, AVG, MIN, MAX, HAVING
👨‍🏫
Aggregates answer "how many?", "what's the total?", "what's the average?" questions. GROUP BY splits your rows into buckets first, then aggregates inside each bucket. The rule: every column in SELECT must either be in GROUP BY or wrapped in an aggregate function.
sql — Core aggregates
-- How many users total?
SELECT COUNT(*) FROM users;

-- Revenue stats
SELECT
    COUNT(*)            AS total_orders,
    SUM(total)          AS total_revenue,
    AVG(total)          AS avg_order_value,
    MIN(total)          AS smallest_order,
    MAX(total)          AS largest_order,
    ROUND(AVG(total), 2) AS avg_rounded
FROM orders
WHERE status != 'cancelled';

-- Revenue per user
SELECT
    u.username,
    COUNT(o.id)    AS order_count,
    SUM(o.total)   AS lifetime_value,
    MAX(o.created_at) AS last_order_date
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.username
ORDER BY lifetime_value DESC NULLS LAST;

HAVING — filtering groups

WHERE filters rows before grouping. HAVING filters groups after aggregation. You can only use aggregate functions in HAVING, not WHERE.

sql — HAVING
-- Only show users who have spent more than $500 total
SELECT
    u.username,
    SUM(o.total) AS total_spent
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE o.status = 'paid'           -- filter rows BEFORE grouping
GROUP BY u.id, u.username
HAVING SUM(o.total) > 500        -- filter groups AFTER aggregation
ORDER BY total_spent DESC;

-- Categories with more than 5 products in stock
SELECT
    category,
    COUNT(*)          AS product_count,
    AVG(price)        AS avg_price,
    SUM(stock_qty)    AS total_stock
FROM products
WHERE is_active = TRUE
GROUP BY category
HAVING COUNT(*) > 5
ORDER BY product_count DESC;

Window Functions — aggregates without collapsing rows

👨‍🏫
This is where PostgreSQL gets seriously powerful. GROUP BY collapses many rows into one. Window functions run aggregates over a set of rows but keep every row in the result. You can rank users, calculate running totals, compare each row to the previous one — all without subqueries.
sql — Window Functions
-- ROW_NUMBER: rank orders by value (1 = highest)
SELECT
    id,
    user_id,
    total,
    ROW_NUMBER() OVER (ORDER BY total DESC) AS rank
FROM orders;

-- PARTITION BY: rank orders within each user
SELECT
    user_id,
    id AS order_id,
    total,
    RANK() OVER (PARTITION BY user_id ORDER BY total DESC) AS rank_per_user
FROM orders;

-- SUM as running total (cumulative revenue)
SELECT
    created_at::date AS day,
    SUM(total) AS daily_revenue,
    SUM(SUM(total)) OVER (ORDER BY created_at::date) AS running_total
FROM orders
WHERE status = 'paid'
GROUP BY created_at::date
ORDER BY day;

-- LAG: compare each order total to the previous one
SELECT
    id,
    total,
    LAG(total) OVER (ORDER BY created_at) AS prev_total,
    total - LAG(total) OVER (ORDER BY created_at) AS change
FROM orders;

-- Top earner per department (classic interview question)
SELECT * FROM (
    SELECT
        username, category, total,
        RANK() OVER (PARTITION BY category ORDER BY total DESC) AS rnk
    FROM orders o JOIN users u ON u.id = o.user_id
) sub WHERE rnk = 1;
FunctionWhat it does
ROW_NUMBER()Unique sequential number per row — no ties. 1,2,3,4...
RANK()Rank with gaps for ties. 1,2,2,4...
DENSE_RANK()Rank without gaps. 1,2,2,3...
LAG(col, n)Value of col from n rows before current row
LEAD(col, n)Value of col from n rows after current row
FIRST_VALUE(col)First value in the window
NTILE(n)Divide rows into n buckets (quartiles, deciles, etc.)
SUM/AVG OVER()Running aggregates over ordered rows
Lesson 05
Subqueries
Queries inside queries — scalar, EXISTS, and derived tables
👨‍🏫
A subquery is just a SELECT inside another SELECT. Before CTEs existed, subqueries were the only way to break complex queries into parts. You still need them constantly. The key skill is knowing where a subquery can live: in WHERE, FROM, SELECT, or HAVING.
sql — Subquery types
-- 1. Scalar subquery in WHERE: orders above average value
SELECT id, total FROM orders
WHERE total > (SELECT AVG(total) FROM orders);

-- 2. IN subquery: users who placed at least one order
SELECT id, username FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);

-- 3. EXISTS (faster than IN for large sets): same query
SELECT id, username FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- 4. NOT EXISTS: users with NO orders
SELECT id, username FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- 5. Derived table in FROM: average order value per user
SELECT u.username, stats.avg_val
FROM users u
JOIN (
    SELECT user_id, AVG(total) AS avg_val
    FROM orders
    GROUP BY user_id
) stats ON stats.user_id = u.id;

-- 6. Scalar subquery in SELECT column (correlated)
SELECT
    id, username,
    (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u;
⚠️ Correlated subquery performance
A correlated subquery in SELECT (example 6 above) runs once per row. For 10,000 users that's 10,000 extra queries. For small tables it's fine, but replace with a LEFT JOIN + COUNT for large tables.
Lesson 06
CTEs — Common Table Expressions
WITH clause: readable, reusable, and recursive
👨‍🏫
CTEs are subqueries with names. Instead of nesting brackets inside brackets, you name each logical step at the top with WITH, then reference them by name. The result is SQL that reads like a story from top to bottom. In 90% of cases, if your query needs a subquery, use a CTE instead — future you will thank you.
sql — CTE basics
-- Without CTE (hard to read)
SELECT u.username, stats.total_spent
FROM users u
JOIN (SELECT user_id, SUM(total) total_spent FROM orders WHERE status='paid' GROUP BY user_id) s
  ON s.user_id = u.id WHERE s.total_spent > 1000;

-- Same query WITH CTE (readable)
WITH paid_order_totals AS (
    SELECT
        user_id,
        SUM(total) AS total_spent,
        COUNT(*)   AS order_count
    FROM orders
    WHERE status = 'paid'
    GROUP BY user_id
)
SELECT
    u.username,
    u.email,
    pot.total_spent,
    pot.order_count
FROM users u
JOIN paid_order_totals pot ON pot.user_id = u.id
WHERE pot.total_spent > 1000
ORDER BY pot.total_spent DESC;
sql — Multiple CTEs chained together
WITH
-- Step 1: get only paid orders
paid_orders AS (
    SELECT * FROM orders WHERE status = 'paid'
),
-- Step 2: total per user from paid orders
user_totals AS (
    SELECT user_id, SUM(total) AS lifetime_value
    FROM paid_orders
    GROUP BY user_id
),
-- Step 3: rank users by value
ranked_users AS (
    SELECT
        user_id,
        lifetime_value,
        NTILE(4) OVER (ORDER BY lifetime_value DESC) AS quartile
    FROM user_totals
)
-- Final query: top quartile users with their info
SELECT u.username, u.email, r.lifetime_value
FROM ranked_users r
JOIN users u ON u.id = r.user_id
WHERE r.quartile = 1
ORDER BY r.lifetime_value DESC;

Recursive CTEs — for hierarchies

sql — Recursive CTE: org chart traversal
-- Walk the org chart from the CEO down to all reports
WITH RECURSIVE org_chart AS (
    -- Base case: the top-level person (no manager)
    SELECT
        id, username, manager_id,
        0               AS depth,
        username::text  AS path
    FROM users
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive case: join to all direct reports
    SELECT
        u.id, u.username, u.manager_id,
        oc.depth + 1,
        oc.path || ' → ' || u.username
    FROM users u
    JOIN org_chart oc ON oc.id = u.manager_id
)
SELECT
    REPEAT('  ', depth) || username AS tree,
    depth,
    path
FROM org_chart
ORDER BY path;
📌 How recursive CTEs work
The base case (before UNION ALL) runs once to get the starting rows. The recursive case (after UNION ALL) runs repeatedly, joining back to the CTE's own output, until no new rows are added. It's like a loop that terminates automatically.
Lesson 07
Indexes Deep Dive
B-tree, GIN, GiST, BRIN, partial, composite — make queries 100x faster
👨‍🏫
An index is a data structure that lets PostgreSQL find rows without scanning the entire table. Without an index on users.email, a query like WHERE email = 'alice@...' reads every single row. With an index, Postgres jumps straight to the answer. The tradeoff: indexes take disk space and slow down writes slightly, so don't index everything blindly.

Index types at a glance

TypeDefault?Best for
B-tree✅ YesEquality, range, ORDER BY, LIKE 'prefix%'. Works for 95% of cases.
HashNoEquality only (= ). Faster than B-tree for pure equality but no range support.
GINNoJSONB, arrays, full-text search. "Does this array contain X?"
GiSTNoGeometric data, PostGIS, full-text search alternatives.
BRINNoVery large tables where values correlate with physical order (timestamps, serial IDs). Tiny index size.
SP-GiSTNoNon-balanced structures: IP ranges, phone numbers, quadtrees.

Creating indexes

sql — Index creation patterns
-- Basic B-tree (default) — speeds up WHERE email = '...'
CREATE INDEX idx_users_email ON users(email);

-- Unique index — same as UNIQUE constraint but explicit
CREATE UNIQUE INDEX idx_users_username ON users(username);

-- Composite index — for queries filtering by both columns
-- Column ORDER matters: put the highest-cardinality column first
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- Partial index — only index rows matching a condition
-- Much smaller and faster than a full index
CREATE INDEX idx_orders_pending ON orders(created_at)
    WHERE status = 'pending';

-- GIN index for JSONB queries
CREATE INDEX idx_users_preferences ON users USING GIN(preferences);

-- GIN index for text search
CREATE INDEX idx_products_fts ON products
    USING GIN(to_tsvector('english', name || ' ' || COALESCE(description,'')));

-- Index on expression
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
-- Now this query uses the index:
-- WHERE LOWER(email) = 'alice@example.com'

-- BRIN for large timestamp tables
CREATE INDEX idx_orders_created_brin ON orders
    USING BRIN(created_at);

-- Create index WITHOUT locking the table (for production!)
CREATE INDEX CONCURRENTLY idx_users_status ON users(status);

-- Drop an index
DROP INDEX idx_users_email;
DROP INDEX CONCURRENTLY idx_users_email; -- production-safe

-- List all indexes on a table
\d users  -- psql: shows all columns, constraints, indexes

When does an index NOT get used?

PatternIndex used?Fix
WHERE LOWER(email) = 'x'❌ No (for regular index)Create expression index on LOWER(email)
WHERE email LIKE '%gmail%'❌ No (leading wildcard)Use full-text search or pg_trgm GIN index
WHERE email LIKE 'alice%'✅ Yes (trailing wildcard only)—
WHERE status != 'active'SometimesPlanner decides — if most rows match, seq scan is faster
WHERE user_id + 1 = 5❌ No (function on column)Rewrite as WHERE user_id = 4

💡 Index strategy — what to index
Always index: foreign key columns, columns in WHERE on large tables, columns in JOIN ON, columns in ORDER BY for paginated queries. Don't index: columns with very few distinct values (boolean, status with 2-3 values on small tables), columns never queried alone.
Lesson 08
EXPLAIN ANALYZE
Read the query plan — find and fix slow queries
👨‍🏫
EXPLAIN ANALYZE is your X-ray machine for slow queries. It shows you exactly how PostgreSQL executed a query: which indexes it used, how many rows it scanned, and how long each step took. Every time a query is slow, run EXPLAIN ANALYZE before guessing. The answer is always in the output.
sql — EXPLAIN vs EXPLAIN ANALYZE
-- EXPLAIN: shows the PLAN without running the query
EXPLAIN
SELECT * FROM users WHERE email = 'alice@example.com';

-- EXPLAIN ANALYZE: RUNS the query and shows actual timings
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'alice@example.com';

-- EXPLAIN (ANALYZE, BUFFERS): also shows cache hit/miss
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 'pending';

Reading EXPLAIN output

Index Scan using idx_users_email on users  (cost=0.29..8.31 rows=1 width=532)
                                              (actual time=0.052..0.054 rows=1 loops=1)
  Index Cond: ((email)::text = 'alice@example.com'::text)
Planning Time: 0.132 ms
Execution Time: 0.079 ms  ← fast! index was used

-------- vs a BAD query (no index) ----------

Seq Scan on users  (cost=0.00..35.50 rows=1 width=532)
                   (actual time=12.4..24.8 rows=1 loops=1)
  Filter: ((LOWER(email))::text = 'alice@example.com')
  Rows Removed by Filter: 9999  ← scanned 10,000 rows to find 1!
Planning Time: 0.1 ms
Execution Time: 24.9 ms  ← 300x slower
Node typeWhat it meansGood or bad?
Seq ScanRead every row in the table (sequential scan)❌ Bad on large tables — add an index
Index ScanUsed an index to find rows, then fetched full rows✅ Good
Index Only ScanAll needed data was in the index — no table fetch✅✅ Best case
Bitmap Heap ScanIndex scan for multiple rows, then batch-fetches from heap✅ Good for range queries
Hash JoinBuild a hash table from one side, probe with other✅ Good for large unsorted joins
Nested LoopFor each row in outer, scan inner (uses index)✅ Good when inner is indexed + small outer
SortExplicit sort step (ORDER BY without an index)⚠️ OK for small, costly for large

Workflow: diagnosing a slow query

1
Run EXPLAIN (ANALYZE, BUFFERS)
Add this in front of your slow query. Never guess — always look at the plan first.
2
Find the most expensive node
Look for the highest actual time. That's where time is being spent. Also look for Seq Scan on large tables and large "Rows Removed by Filter" numbers.
3
Add a targeted index
Index the column(s) in the expensive filter or join. Use CREATE INDEX CONCURRENTLY in production so you don't lock the table.
4
Re-run EXPLAIN ANALYZE
Confirm the planner now uses your new index and execution time dropped. If not, ANALYZE tablename to update statistics and try again.
💡 Use explain.dalibo.com
Paste your EXPLAIN output at explain.dalibo.com for a beautiful visual plan — much easier to read than raw text, especially for complex multi-join queries.
