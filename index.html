Lesson 01
What is PostgreSQL?
Understanding what you're about to learn and why it matters
👨‍🏫
Think of a database like a filing cabinet. Inside the cabinet are folders (tables). Each folder has papers (rows) with info written in labeled sections (columns). PostgreSQL is the smartest, most powerful filing cabinet you can get — and it's completely free.

PostgreSQL (pronounced post-gress-Q-L, often just called Postgres) is an open-source relational database management system (RDBMS). It stores data in tables that relate to each other — just like a spreadsheet, but far more powerful.

It was born at UC Berkeley in 1986 as POSTGRES, went open-source in 1994, and has been battle-tested for nearly 40 years. Today it powers companies like Instagram, Twitch, Spotify, Reddit, and GitHub.

What does PostgreSQL actually do?

📦
Stores data reliably
Keeps your data safe on disk, even during crashes, thanks to Write-Ahead Logging (WAL).
🔍
Answers questions fast
You ask "give me all users who signed up this month" and Postgres returns the answer in milliseconds.
🔒
Controls who sees what
Roles and permissions let you decide which users or apps can read, write, or delete data.
🔗
Links related data
A user row can link to many order rows. Postgres enforces these links so you can't accidentally orphan data.
⚡
Handles millions of rows
With the right indexes, Postgres searches a table of 100 million rows in milliseconds.
🧩
Stores any kind of data
Text, numbers, dates, JSON, arrays, geo coordinates, images — Postgres handles it all natively.
📌 Key Term
A relational database stores data in tables (like spreadsheets) where tables can reference each other. You interact with it using SQL — Structured Query Language.
Lesson 02
Why PostgreSQL?
How it compares to MySQL, SQLite, and NoSQL
👨‍🏫
You'll hear about MySQL, MongoDB, SQLite — here's how to think about the choice. Most professional backend developers use PostgreSQL as their go-to because it's the most featureful SQL database that's also free. SQLite is for tiny projects. MySQL is older and less capable. MongoDB is for when your data has no structure at all.

🐘 PostgreSQL — Choose when...

  • Building a real app (API, web app, SaaS)
  • You need JSON + SQL in the same place
  • You want advanced features (window fns, CTEs)
  • Correctness and data integrity matter
  • You want full ACID compliance

🦈 MySQL — Choose when...

  • You're maintaining a legacy app
  • WordPress / PHP stack (it was default)
  • Simple read-heavy workloads
  • Less true ACID compliance (old versions)

🪶 SQLite — Choose when...

  • Mobile apps, embedded systems
  • Local dev, testing, prototyping
  • No server needed (single file on disk)
  • Very low concurrency requirements

🍃 MongoDB — Choose when...

  • Data is truly schema-less and varies wildly
  • Storing documents, not rows
  • Sacrificing relations for scale (rare)
  • Note: Postgres JSON is often good enough

PostgreSQL's Superpowers

FeatureWhat it meansMySQL has it?
JSONBStore and query JSON like a document databasePartial
Window FunctionsRank, running totals, lag/lead without subqueriesYes (v8+)
Full-Text SearchBuilt-in search engine, no Elasticsearch neededBasic only
ArraysStore multiple values in a single columnNo
Partial IndexesIndex only rows matching a conditionNo
Table InheritanceOOP-style table hierarchiesNo
PostGISGeo-spatial queries (distance, polygons, maps)No
True ACIDFull transaction guarantees always onDepends on engine
💡 Bottom line
If you're learning one database for backend development, PostgreSQL is the right answer. It will be relevant at every company you ever work at.
Lesson 03
Installation
Get PostgreSQL running on your machine in under 5 minutes
👨‍🏫
We're going to install PostgreSQL and confirm it's running before moving on. Don't skip this — everything else requires a working installation. Pick your operating system below.
🍺 Homebrew (Recommended)
# Install Homebrew first if you don't have it
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Install PostgreSQL
brew install postgresql@16

# Add to PATH (copy/paste the line brew tells you, or run this)
echo 'export PATH="/opt/homebrew/opt/postgresql@16/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

# Start PostgreSQL as a background service
brew services start postgresql@16

# Verify it's running
pg_isready
✅ Success looks like
/tmp/.s.PGSQL.5432 - accepting connections — that means Postgres is live on port 5432.
🐧 Ubuntu / Debian
# Update package list
sudo apt update

# Install PostgreSQL
sudo apt install -y postgresql postgresql-contrib

# Start and enable the service
sudo systemctl start postgresql
sudo systemctl enable postgresql

# Check status
sudo systemctl status postgresql

# Connect as the default postgres superuser
sudo -u postgres psql
🪟 Windows
# Option 1: Official installer (easiest)
# Download from: https://www.postgresql.org/download/windows/
# Run the installer, pick a password for postgres user, default port 5432

# Option 2: WSL2 (recommended for developers)
# Install WSL2, then follow the Linux steps above

# After install, open "SQL Shell (psql)" from Start menu
# Or add PostgreSQL\bin to your PATH and run:
psql -U postgres

Connecting for the first time

bash — First connection
# On macOS (Homebrew): your username is the default superuser
psql postgres

# On Linux / Windows: connect as the postgres superuser
psql -U postgres

# Connect to a specific database on a specific host
psql -h localhost -p 5432 -U myuser -d mydb

# If successful, you'll see the psql prompt:

PostgreSQL's bin directory isn't in your PATH. On macOS with Homebrew: run brew link postgresql@16 or add it manually to your PATH in ~/.zshrc.

Lesson 04
psql — The Command-Line Interface
Master the terminal tool every PostgreSQL user needs to know
👨‍🏫
psql is your direct line to PostgreSQL. Think of it as a conversation — you type a SQL question, Postgres gives you an answer. Every database professional uses psql. The backslash commands (\d, \l, etc.) are psql-specific shortcuts that don't end with a semicolon.

Essential psql Commands

\lList all databases
\c dbnameConnect to a database
\dtList all tables in current DB
\d tablenameDescribe a table's structure
\dnList all schemas
\duList all users/roles
\i file.sqlExecute a SQL file
\eOpen query in text editor
\timingShow query execution time
\xToggle expanded (vertical) output
\?Help for psql commands
\h SELECTSQL command help
\qQuit psql
↑ ↓Navigate command history
psql — A real session walkthrough
-- You just connected. The prompt shows you're in the "postgres" database
postgres=#

-- List all databases
postgres=# \l
--  Name    |  Owner   | Encoding | ...
-- ---------+----------+----------+----
--  postgres | postgres | UTF8     | ...
--  template0| postgres | UTF8     | ...

-- Create a new database and connect to it
postgres=# CREATE DATABASE myapp;
CREATE DATABASE
postgres=# \c myapp
-- You are now connected to database "myapp" as user "postgres".
myapp=#

-- Check what tables exist (none yet)
myapp=# \dt
-- Did not find any relations.

-- Try \x for nicer output on wide tables
myapp=# \x
-- Expanded display is on.
⚠️ Don't forget the semicolon!
SQL queries in psql MUST end with ;. If you hit Enter without one, psql waits for more input (the prompt changes to -#). Just type ; and hit Enter to finish, or \q to cancel.

Useful psql tricks

psql — Power user tips
-- Run a single query from the shell without entering psql
psql -U postgres -d myapp -c "SELECT version();"

-- Run a SQL file
psql -U postgres -d myapp -f setup.sql

-- Output query results to a CSV file
psql -U postgres -d myapp -c "\copy users TO 'users.csv' CSV HEADER"

-- Check what version of Postgres is installed
postgres=# SELECT version();
-- PostgreSQL 16.2 on aarch64-apple-darwin...

-- See current connection info
postgres=# \conninfo
-- You are connected to database "postgres" as user "postgres" via socket...
Lesson 05
How PostgreSQL Works Internally
A mental model that will help you debug and optimize
👨‍🏫
You don't need to memorize this, but you DO need a mental model. When you understand how Postgres processes a query, you'll know why indexes matter, why certain queries are slow, and how to fix them. This pays dividends for your whole career.

The journey of a query

1
Client sends a query
Your app (or psql) sends SQL text like SELECT * FROM users WHERE id = 5 over a TCP connection to the Postgres server on port 5432.
2
Parser — checks syntax
Postgres parses your SQL into a parse tree. If you have a typo or missing keyword, it errors here: "syntax error at or near..."
3
Rewriter — applies rules
Transforms the query. For example, a VIEW reference gets expanded into its underlying SELECT statement here.
4
Planner / Optimizer — chooses the best plan
The most important step. Postgres estimates many ways to fetch the data (sequential scan vs. index scan vs. hash join) and picks the cheapest one. This is what EXPLAIN shows you.
5
Executor — runs the plan
Fetches rows from disk (or the shared buffer cache), applies filters, sorts, and returns results to the client.

Key architectural concepts

ConceptPlain EnglishWhy you care
Shared BuffersRAM cache for frequently-used data pagesReads from RAM are ~1000x faster than disk
WAL (Write-Ahead Log)Every change is logged to disk before it's appliedCrash safety + replication
MVCCMultiple readers never block writers (each gets their own "snapshot")High concurrency without locks
VacuumingBackground cleanup of old row versionsPrevents table bloat over time
pg_hba.confControls who can connect and howSecurity configuration
postgresql.confMain config file (memory, connections, logging)Performance tuning
📌 MVCC — Multi-Version Concurrency Control
When you read a row and someone else is updating it at the same time, you see the old version, they write the new version. No lock needed. This is why Postgres handles many simultaneous users so well.
Lesson 06
Databases, Schemas & Users
The hierarchy of organization in PostgreSQL
👨‍🏫
Here's the hierarchy: One PostgreSQL server can have many databases. Each database has many schemas. Each schema has many tables. Think of it like: building → floors → rooms → furniture.

The hierarchy

Structure
PostgreSQL Server (port 5432)
├── Database: myapp
│   ├── Schema: public        ← default schema
│   │   ├── Table: users
│   │   ├── Table: orders
│   │   └── Table: products
│   └── Schema: analytics
│       ├── Table: events
│       └── Table: funnels
├── Database: myapp_test
│   └── Schema: public
│       └── (test tables)
└── Database: postgres        ← default system database

Creating and managing databases

sql — Database commands
-- Create a database
CREATE DATABASE myapp;

-- Create with specific settings
CREATE DATABASE myapp
    OWNER = postgres
    ENCODING = 'UTF8'
    LC_COLLATE = 'en_US.UTF-8'
    LC_CTYPE = 'en_US.UTF-8';

-- List all databases
\l    -- psql shortcut
SELECT datname FROM pg_database;  -- SQL way

-- Switch to a database (psql only)
\c myapp

-- Rename a database
ALTER DATABASE myapp RENAME TO myapp_v2;

-- Delete a database (CAREFUL: drops everything inside)
DROP DATABASE myapp;
DROP DATABASE IF EXISTS myapp;  -- safe version

Schemas

A schema is a namespace inside a database. By default, everything goes into the public schema. Use schemas to separate concerns — e.g., app schema for business tables, analytics for reporting tables.

sql — Schema commands
-- Create a schema
CREATE SCHEMA analytics;
CREATE SCHEMA IF NOT EXISTS analytics;

-- Create a table in a specific schema
CREATE TABLE analytics.events (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

-- The search_path controls which schema is the default
SET search_path TO analytics, public;

-- List schemas
\dn

Creating users and roles

sql — Users & Roles
-- Create a user (a role that can log in)
CREATE USER appuser WITH PASSWORD 'securepassword123';

-- Grant all privileges on a database
GRANT ALL PRIVILEGES ON DATABASE myapp TO appuser;

-- Grant access to a schema
GRANT USAGE ON SCHEMA public TO appuser;

-- Grant SELECT/INSERT/UPDATE/DELETE on all current tables
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO appuser;

-- Grant privileges on future tables too (important!)
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO appuser;

-- Make a read-only user
CREATE USER readonly_user WITH PASSWORD 'readonlypass';
GRANT CONNECT ON DATABASE myapp TO readonly_user;
GRANT USAGE ON SCHEMA public TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;
⚠️ Security Best Practice
Never use the postgres superuser in your application. Create a dedicated user with only the permissions your app needs. If that user gets compromised, the damage is limited.
Lesson 07
PostgreSQL Data Types
PostgreSQL has the richest type system of any SQL database — know what to use
👨‍🏫
Choosing the right data type matters for two reasons: correctness and performance. A column storing prices should be NUMERIC, not FLOAT (floats have rounding errors — never use them for money). A column storing a user's ID should be INTEGER or BIGINT, not VARCHAR.

Numeric Types

SMALLINT
Whole number. Range: –32,768 to 32,767. Use for small counters, ages, ratings.
age SMALLINT
INTEGER / INT
Whole number. Range: –2.1B to 2.1B. The most common numeric type.
quantity INT
BIGINT
Whole number. Range: ±9.2 quintillion. Use for user IDs, counts that could get huge.
user_id BIGINT
NUMERIC(p, s) / DECIMAL
Exact decimal. Always use this for money. p = total digits, s = decimal places.
price NUMERIC(10, 2) → 99999999.99
REAL / FLOAT4
Approximate float, 6 decimal digits. Use for scientific data — NOT money.
temperature REAL
DOUBLE PRECISION / FLOAT8
Approximate float, 15 decimal digits. More precise than REAL but still not exact.
latitude DOUBLE PRECISION
SERIAL PG
Auto-incrementing integer (1, 2, 3…). Shorthand for creating a sequence. Use for primary keys.
id SERIAL PRIMARY KEY
BIGSERIAL PG
Auto-incrementing BIGINT. Use when you expect more than 2.1 billion rows.
id BIGSERIAL PRIMARY KEY

Text Types

VARCHAR(n)
Variable-length string, max n characters. Use when you have a known max length (name, email).
email VARCHAR(255)
CHAR(n)
Fixed-length string, always padded to n chars. Rarely useful — use VARCHAR instead.
country_code CHAR(2)
TEXT PG
Unlimited length string. In Postgres, TEXT and VARCHAR(n) have identical performance. Use TEXT freely.
bio TEXT
💡 PostgreSQL-specific tip
In most databases, VARCHAR is faster than TEXT. In PostgreSQL, they are exactly the same internally. Use TEXT when there's no meaningful max length, and VARCHAR(n) only when you want to enforce a limit as a constraint.

Date & Time Types

DATE
Date only: year, month, day. Range: 4713 BC to 5874897 AD.
birth_date DATE → '1995-07-15'
TIME
Time of day only, no timezone. Range: 00:00:00 to 24:00:00.
opens_at TIME → '09:00:00'
TIMESTAMP
Date + time, no timezone. Stores what you put in. Careful with multi-timezone apps.
created_at TIMESTAMP → '2024-01-15 14:30:00'
TIMESTAMPTZ PG
Date + time + timezone awareness. Recommended for almost all timestamps. Stored in UTC, shown in local time.
created_at TIMESTAMPTZ
INTERVAL PG
A span of time. Can add/subtract from dates. Very useful for "30 days ago" type queries.
duration INTERVAL → '2 hours 30 minutes'

Boolean & Special Types

BOOLEAN
TRUE or FALSE. Also accepts 'yes'/'no', 'on'/'off', '1'/'0' on input.
is_active BOOLEAN DEFAULT TRUE
UUID PG
Universally unique identifier. 128-bit. Great for distributed systems where you need globally unique IDs.
id UUID DEFAULT gen_random_uuid()
JSONB PG
Binary JSON. Indexed, queryable. The #1 reason developers switch from MySQL to Postgres.
metadata JSONB
ARRAY PG
Array of any type. Store multiple values in one column without a join table (for simple cases).
tags TEXT[]
ENUM PG
Custom type with a fixed set of values. Faster and safer than using VARCHAR for statuses.
status order_status → 'pending','shipped'
BYTEA
Raw binary data. Use for small files, hashes, cryptographic keys.
avatar BYTEA
sql — Creating a custom ENUM type
-- Define a custom enum type first
CREATE TYPE order_status AS ENUM (
    'pending', 'processing', 'shipped', 'delivered', 'cancelled'
);

-- Use it in a table
CREATE TABLE orders (
    id     SERIAL PRIMARY KEY,
    status order_status NOT NULL DEFAULT 'pending'
);

-- Adding a value to an existing enum
ALTER TYPE order_status ADD VALUE 'refunded';
Lesson 08
Your First Table
CREATE TABLE — designing it right the first time
👨‍🏫
We're going to build a realistic users table — the table almost every app has. I'll show you what each column does and why. We'll also look at the ALTER TABLE commands you'll use when requirements change (because they always do).
sql — CREATE TABLE with every important feature
-- First, connect to your database
\c myapp

-- Create a realistic users table
CREATE TABLE users (
    -- Primary key: auto-incrementing integer
    id            BIGSERIAL    PRIMARY KEY,

    -- Basic info
    username      VARCHAR(50)  NOT NULL UNIQUE,
    email         VARCHAR(255) NOT NULL UNIQUE,
    password_hash TEXT         NOT NULL,

    -- Optional profile info
    first_name    VARCHAR(100),
    last_name     VARCHAR(100),
    bio           TEXT,
    avatar_url    TEXT,

    -- Status as an enum
    status        VARCHAR(20)  NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'suspended', 'deleted')),

    -- Boolean flags
    is_verified   BOOLEAN      NOT NULL DEFAULT FALSE,
    is_admin      BOOLEAN      NOT NULL DEFAULT FALSE,

    -- Flexible metadata as JSONB
    preferences   JSONB        DEFAULT '{}'::jsonb,

    -- Timestamps (always use TIMESTAMPTZ!)
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    last_login_at TIMESTAMPTZ
);

-- Verify the table structure
\d users

Understanding each constraint

ConstraintWhat it doesWhat happens on violation
PRIMARY KEYUniquely identifies every row. Implicitly NOT NULL + UNIQUE.Error: duplicate key value
NOT NULLColumn must always have a value — NULL is rejected.Error: null value in column violates not-null constraint
UNIQUEEvery value in this column must be different across all rows.Error: duplicate key value violates unique constraint
DEFAULTIf no value given on INSERT, use this value instead.No error — uses default silently
CHECK(...)Custom condition every inserted/updated value must satisfy.Error: new row violates check constraint
REFERENCESForeign key — value must exist in another table.Error: violates foreign key constraint

Altering tables after creation

sql — ALTER TABLE
-- Add a new column
ALTER TABLE users
ADD COLUMN phone_number VARCHAR(20);

-- Add a column with a default (backfills existing rows)
ALTER TABLE users
ADD COLUMN login_count INT NOT NULL DEFAULT 0;

-- Rename a column
ALTER TABLE users
RENAME COLUMN avatar_url TO profile_image_url;

-- Change a column's data type
ALTER TABLE users
ALTER COLUMN phone_number TYPE VARCHAR(30);

-- Add a NOT NULL constraint (existing NULL rows will fail!)
ALTER TABLE users
ALTER COLUMN phone_number SET NOT NULL;

-- Remove NOT NULL constraint
ALTER TABLE users
ALTER COLUMN phone_number DROP NOT NULL;

-- Drop a column
ALTER TABLE users
DROP COLUMN phone_number;

-- Rename the table
ALTER TABLE users RENAME TO app_users;
🚨 Production warning
ALTER TABLE on large tables can lock the entire table in older Postgres versions. Adding a column with a DEFAULT in Postgres 11+ is safe and instant. Adding a NOT NULL constraint requires a full table scan. Always test schema changes on a copy of prod data first.
Lesson 09
CRUD Operations
INSERT, SELECT, UPDATE, DELETE — the four operations you'll use every day
👨‍🏫
CRUD = Create, Read, Update, Delete. Every operation you'll ever do on data fits into one of these four buckets. Let's work with the users table we just created and see each one in action. Follow along in your psql session.

INSERT — Create rows

sql — INSERT
-- Insert a single user
INSERT INTO users (username, email, password_hash, first_name, last_name)
VALUES ('alice_j', 'alice@example.com', '$2b$10$hashedvalue', 'Alice', 'Johnson');

-- Insert and get back the new row (RETURNING is PostgreSQL-specific ✨)
INSERT INTO users (username, email, password_hash)
VALUES ('bob_smith', 'bob@example.com', '$2b$10$hashedvalue2')
RETURNING id, username, created_at;

-- Insert multiple rows at once
INSERT INTO users (username, email, password_hash, is_admin)
VALUES
    ('carol_d',  'carol@example.com',  '$2b$10$hash3', FALSE),
    ('david_k',  'david@example.com',  '$2b$10$hash4', FALSE),
    ('superadmin','admin@example.com', '$2b$10$hash5', TRUE);

-- Insert with JSON preferences
INSERT INTO users (username, email, password_hash, preferences)
VALUES (
    'eve_r',
    'eve@example.com',
    '$2b$10$hash6',
    '{"theme": "dark", "language": "en", "notifications": true}'::jsonb
)
RETURNING *;

-- Upsert: insert OR update if username already exists (PostgreSQL 9.5+)
INSERT INTO users (username, email, password_hash)
VALUES ('alice_j', 'alice_new@example.com', '$2b$10$newhash')
ON CONFLICT (username)
DO UPDATE SET
    email = EXCLUDED.email,
    updated_at = NOW();
💡 RETURNING — PostgreSQL superpower
RETURNING is a PostgreSQL extension that lets you get the inserted/updated/deleted row data back immediately. In your app code you can do INSERT ... RETURNING id to get the new row's ID without a second query.

SELECT — Read rows

sql — SELECT
-- Get all users
SELECT * FROM users;

-- Get specific columns
SELECT id, username, email, created_at FROM users;

-- Filter with WHERE
SELECT * FROM users
WHERE is_admin = TRUE;

-- Text search (case-sensitive)
SELECT * FROM users
WHERE email LIKE '%@example.com';

-- Case-insensitive search (PostgreSQL ILIKE ✨)
SELECT * FROM users
WHERE username ILIKE 'alice%';

-- Order results
SELECT id, username, created_at FROM users
ORDER BY created_at DESC;

-- Pagination
SELECT * FROM users
ORDER BY id
LIMIT 10 OFFSET 20;  -- page 3 of 10-per-page

-- Count rows
SELECT COUNT(*) FROM users;
SELECT COUNT(*) FROM users WHERE is_verified = TRUE;

-- Query JSONB field ✨
SELECT username, preferences->>'theme' AS theme
FROM users
WHERE preferences->>'theme' = 'dark';

-- NULL checks
SELECT * FROM users WHERE last_login_at IS NULL;    -- never logged in
SELECT * FROM users WHERE last_login_at IS NOT NULL; -- has logged in

-- Computed columns
SELECT
    id,
    first_name || ' ' || last_name AS full_name,
    UPPER(email) AS email_upper,
    NOW() - created_at AS account_age
FROM users;

UPDATE — Modify rows

sql — UPDATE
-- Update a specific user by ID
UPDATE users
SET is_verified = TRUE,
    updated_at  = NOW()
WHERE id = 1;

-- Update and see what changed (RETURNING ✨)
UPDATE users
SET last_login_at = NOW()
WHERE username = 'alice_j'
RETURNING id, username, last_login_at;

-- Update multiple rows at once
UPDATE users
SET status = 'suspended'
WHERE is_verified = FALSE
  AND created_at < NOW() - INTERVAL '30 days';

-- Update a JSONB field
UPDATE users
SET preferences = preferences || '{"theme": "light"}'::jsonb
WHERE id = 5;
🚨 Always use WHERE with UPDATE and DELETE
UPDATE users SET status = 'deleted'; without a WHERE clause updates every single row in the table. This is one of the most common catastrophic mistakes. Always check your WHERE clause first with a SELECT.

DELETE — Remove rows

sql — DELETE
-- Always SELECT first to verify what you'll delete
SELECT * FROM users WHERE id = 3;

-- Then delete it
DELETE FROM users WHERE id = 3;

-- Delete and confirm what was removed (RETURNING ✨)
DELETE FROM users
WHERE status = 'deleted'
RETURNING id, username, email;

-- Safe pattern: use a transaction so you can rollback
BEGIN;
DELETE FROM users WHERE created_at < '2022-01-01';
-- Check how many rows were affected
-- If happy: COMMIT; If not: ROLLBACK;
ROLLBACK;  -- ← change to COMMIT when you're sure
💡 Pro tip — Soft deletes
In real apps, most teams don't actually DELETE rows. Instead they set deleted_at = NOW() (a "soft delete"). This preserves history and allows recovery. Then filter with WHERE deleted_at IS NULL in all queries.
🧠 Quick check: Which command would you use to get back the newly inserted row's ID without running a second SELECT?
✅ B is correct. RETURNING is a PostgreSQL extension that returns the specified columns from the inserted row immediately. Note: SELECT lastval() also works for serial IDs but is a separate query and session-specific — RETURNING is safer and more explicit.
