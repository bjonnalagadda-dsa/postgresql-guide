Lesson 01
Table Partitioning
Split massive tables into manageable pieces — keep queries fast at scale
👨‍🏫
When your orders table has 500 million rows, even an indexed query gets slow. Partitioning splits one logical table into many physical sub-tables called partitions. A query for "orders this month" only scans November's partition — not all 500M rows. PostgreSQL handles this transparently. Your app still queries orders like normal.

Three partitioning strategies

📅 RANGE

Split by value ranges — dates, IDs. Most common. Orders by month, events by year, logs by day.

📋 LIST

Split by explicit values. Orders by country, events by region. Good when you have a fixed set of categories.

#️⃣ HASH

Split by hash of a column. Even distribution across N partitions. Good when no natural range/list exists.

sql — RANGE partitioning by month
-- Step 1: Create the parent (partitioned) table
CREATE TABLE orders_partitioned (
    id         BIGSERIAL,
    user_id    BIGINT NOT NULL,
    total      NUMERIC(12,2),
    status     VARCHAR(20),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);  -- partition key

-- Step 2: Create child partitions (one per month)
CREATE TABLE orders_2024_01 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE orders_2024_02 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

CREATE TABLE orders_2024_03 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');

-- A default partition catches any rows that don't match other partitions
CREATE TABLE orders_default PARTITION OF orders_partitioned DEFAULT;

-- Step 3: Add indexes ON the partitions (or on the parent — auto-applies)
CREATE INDEX ON orders_partitioned(user_id);
CREATE INDEX ON orders_partitioned(status, created_at);

-- Step 4: INSERT works exactly as before — auto-routes to correct partition
INSERT INTO orders_partitioned (user_id, total, status, created_at)
VALUES (1, 99.99, 'paid', '2024-01-15');

-- Step 5: Query just scans the relevant partition (partition pruning)
EXPLAIN
SELECT * FROM orders_partitioned
WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';
-- You'll see: Seq Scan on orders_2024_01 (only that partition!)
sql — LIST and HASH partitioning
-- LIST: partition by country
CREATE TABLE customers (
    id      BIGSERIAL,
    name    TEXT,
    country CHAR(2) NOT NULL
) PARTITION BY LIST(country);

CREATE TABLE customers_us PARTITION OF customers
    FOR VALUES IN ('US');
CREATE TABLE customers_eu PARTITION OF customers
    FOR VALUES IN ('DE', 'FR', 'GB', 'IT', 'ES');
CREATE TABLE customers_other PARTITION OF customers DEFAULT;

-- HASH: spread evenly across 4 partitions by user_id
CREATE TABLE events (
    id      BIGSERIAL,
    user_id BIGINT NOT NULL,
    name    TEXT
) PARTITION BY HASH(user_id);

CREATE TABLE events_p0 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE events_p1 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE events_p2 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE events_p3 PARTITION OF events
    FOR VALUES WITH (MODULUS 4, REMAINDER 3);

Partition management

sql — Attach, detach, drop old partitions
-- Detach a partition (keeps the table, just no longer part of parent)
ALTER TABLE orders_partitioned
DETACH PARTITION orders_2024_01;

-- Drop an old partition instantly — WAY faster than DELETE
DROP TABLE orders_2024_01;

-- Attach a pre-existing table as a new partition
ALTER TABLE orders_partitioned
ATTACH PARTITION orders_2024_12
FOR VALUES FROM ('2024-12-01') TO ('2025-01-01');

-- List all partitions
SELECT inhrelid::regclass AS partition_name,
       pg_size_pretty(pg_relation_size(inhrelid)) AS size
FROM   pg_inherits
WHERE  inhparent = 'orders_partitioned'::regclass;
💡 When to partition
Start thinking about partitioning when a single table exceeds 100GB or 100M rows. Below that, good indexes are usually enough. The biggest benefit: dropping old partitions (DROP TABLE orders_2022_01) is instantaneous — deleting 100M rows is not.
Lesson 02
Replication & High Availability
Streaming replication, read replicas, failover — keeping your DB alive
👨‍🏫
A single database server is a single point of failure. If it crashes, your app goes down. Replication solves this by keeping one or more standby servers that are exact copies of the primary. If primary dies, a standby takes over in seconds. Read replicas also let you offload heavy read queries to separate servers.
┌──────────────────────────────────────────────────────────────┐ │ Your Application │ └────────────────┬─────────────────────────┬───────────────────┘ │ writes │ reads (load balanced) ▼ ▼ ┌─────────────────┐ ┌────────────────────────────────┐ │ PRIMARY (RW) │───▶│ STANDBY / REPLICA (RO) │ │ postgres:5432 │ │ postgres:5433 │ └─────────────────┘ └────────────────────────────────┘ │ WAL stream ↑ streaming replication └──────────────────────────┘ Changes flow via WAL. Replica lag is typically < 1 second.

Types of replication

TypeWhat replicatesUse case
Streaming (Physical)Every byte — full copy of the entire clusterHigh availability, hot standby, read replicas
LogicalRow-level changes for selected tablesSelective replication, migrations, data pipelines
SynchronousCommit only after standby confirms receiptZero data loss — slower writes
AsynchronousCommit immediately, standby catches upDefault — tiny risk of data loss on crash
bash + sql — Setting up streaming replication
════ ON THE PRIMARY SERVER ════════════════════════════════════

-- Create a dedicated replication user
CREATE USER replicator
    WITH REPLICATION
    ENCRYPTED PASSWORD 'strong_password_here';

-- In postgresql.conf:
wal_level = replica           -- or 'logical' for logical replication
max_wal_senders = 5           -- max simultaneous standbys
wal_keep_size = 1GB           -- keep this much WAL for slow standbys
hot_standby = on              -- allow reads on standby

-- In pg_hba.conf (allow replicator to connect):
host  replication  replicator  10.0.0.2/32  scram-sha-256

-- Reload config
SELECT pg_reload_conf();

════ ON THE STANDBY SERVER ════════════════════════════════════

# Take a base backup from primary to initialize standby
pg_basebackup \
  -h primary_host \
  -U replicator \
  -D /var/lib/postgresql/16/main \
  -P -Xs -R
# -R writes the recovery config automatically
# -Xs streams WAL during backup

# Start the standby — it auto-follows the primary
systemctl start postgresql

════ CHECK REPLICATION STATUS ═════════════════════════════════

-- On primary: see all standbys and their lag
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(sent_lsn, replay_lsn)
    ) AS replication_lag
FROM pg_stat_replication;

Logical Replication — replicate selected tables

sql — Logical replication
-- On the SOURCE database: create a publication
CREATE PUBLICATION my_pub FOR TABLE users, orders;

-- Or replicate all tables
CREATE PUBLICATION all_tables_pub FOR ALL TABLES;

-- On the DESTINATION database: create a subscription
CREATE SUBSCRIPTION my_sub
    CONNECTION 'host=source_host dbname=myapp user=replicator password=xxx'
    PUBLICATION my_pub;

-- Check subscription status
SELECT * FROM pg_stat_subscription;
📌 Managed options — skip the setup pain
In production, most teams use managed PostgreSQL: AWS RDS/Aurora, Google Cloud SQL, Supabase, Neon, or Railway. These handle replication, backups, and failover automatically. Know how replication works conceptually — but don't run your own unless you have a DBA.
Lesson 03
Backup & Restore
pg_dump, pg_restore, pg_basebackup — never lose data
👨‍🏫
Backups are only valuable if you've tested restoring from them. PostgreSQL gives you two main tools: pg_dump for logical backups (SQL or binary of a single database) and pg_basebackup for physical backups (the entire data directory). Know both. Test your restore process before you need it in an emergency.
bash — pg_dump: logical backups
═══ BACKUP ═══════════════════════════════════════════════════

# Plain SQL dump (human-readable, good for small DBs)
pg_dump -U postgres -d myapp > myapp_backup.sql

# Custom format (compressed, allows selective restore)
pg_dump -U postgres -d myapp -Fc -f myapp_backup.dump

# With compression level (0=none, 9=max)
pg_dump -U postgres -d myapp -Fc -Z9 -f myapp_backup.dump

# Dump only specific tables
pg_dump -U postgres -d myapp -t users -t orders -Fc -f partial.dump

# Dump schema only (no data)
pg_dump -U postgres -d myapp --schema-only -f schema.sql

# Dump all databases (roles, tablespaces, all DBs)
pg_dumpall -U postgres > full_cluster.sql

═══ RESTORE ═══════════════════════════════════════════════════

# Restore SQL dump
psql -U postgres -d myapp < myapp_backup.sql

# Restore custom format (parallel restore with 4 workers = much faster)
pg_restore -U postgres -d myapp -j 4 myapp_backup.dump

# Restore only specific tables from a dump
pg_restore -U postgres -d myapp -t users myapp_backup.dump

# Create the target database first
createdb -U postgres myapp_restored
pg_restore -U postgres -d myapp_restored myapp_backup.dump
bash — Automated daily backup script
#!/bin/bash
# Save as /usr/local/bin/pg_backup.sh — run via cron

DB_NAME="myapp"
DB_USER="postgres"
BACKUP_DIR="/backups/postgres"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
KEEP_DAYS=30  # delete backups older than this

mkdir -p "$BACKUP_DIR"

# Take the backup
pg_dump -U "$DB_USER" -Fc -Z6 "$DB_NAME" \
    > "$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.dump"

if [ $? -eq 0 ]; then
    echo "✅ Backup succeeded: ${DB_NAME}_${TIMESTAMP}.dump"
else
    echo "❌ Backup FAILED!" >&2
    exit 1
fi

# Delete old backups
find "$BACKUP_DIR" -name "*.dump" -mtime +$KEEP_DAYS -delete
echo "Cleaned up backups older than $KEEP_DAYS days"

# Add to crontab: run daily at 2am
# crontab -e
# 0 2 * * * /usr/local/bin/pg_backup.sh >> /var/log/pg_backup.log 2>&1

Point-In-Time Recovery (PITR)

PITR lets you restore a database to any moment in the past — not just your last backup. It combines a physical base backup with WAL archive files. This is the gold standard for production databases.

conf — Enable WAL archiving for PITR
# In postgresql.conf:
archive_mode = on
archive_command = 'cp %p /wal_archive/%f'  # or use aws s3 cp, etc.
restore_command = 'cp /wal_archive/%f %p'

# Take a base backup:
pg_basebackup -U postgres -D /backup/base -Xs -P

# To restore to a specific time, create recovery.conf (or recovery params):
recovery_target_time = '2024-06-01 14:30:00'
recovery_target_action = 'promote'
🚨 The 3-2-1 backup rule
Keep 3 copies of your data, on 2 different storage types, with 1 offsite. And: test your restore every month. A backup you've never restored is a backup you can't trust.
Lesson 04
Configuration Tuning
postgresql.conf — squeeze maximum performance out of your server
👨‍🏫
PostgreSQL's defaults are conservative — designed to work on any hardware. On a real server with 32GB RAM, the defaults leave 95% of memory unused. Tuning postgresql.conf properly can make your database 3–10x faster without changing a single query. The most important settings are memory-related.
conf — Key settings for a 32GB RAM production server
═══ MEMORY ════════════════════════════════════════════════════
# shared_buffers: PostgreSQL's own page cache
# Rule: 25% of total RAM
shared_buffers = 8GB

# work_mem: memory per sort/hash operation PER QUERY PER NODE
# Careful: 10 connections × 5 nodes × work_mem = total used
# Rule: (RAM - shared_buffers) / (max_connections × 3)
work_mem = 64MB

# maintenance_work_mem: for VACUUM, CREATE INDEX, CLUSTER
maintenance_work_mem = 2GB

# effective_cache_size: estimate of OS + PG cache (planner hint)
# Rule: 75% of total RAM
effective_cache_size = 24GB

═══ CONNECTIONS ═══════════════════════════════════════════════
# Each connection uses ~5-10MB RAM. Keep this low, use PgBouncer.
max_connections = 200

═══ WAL / CHECKPOINTS ═════════════════════════════════════════
# Larger WAL = fewer checkpoints = better write throughput
wal_buffers = 64MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB
min_wal_size = 1GB

═══ PARALLELISM ═══════════════════════════════════════════════
# Allow queries to use multiple CPU cores
max_parallel_workers_per_gather = 4
max_parallel_workers = 8
max_worker_processes = 16

═══ PLANNER ═══════════════════════════════════════════════════
# SSDs: lower these so planner prefers index scans
random_page_cost = 1.1    # 4.0 for HDDs, 1.1 for SSDs
effective_io_concurrency = 200  # SSDs handle many concurrent reads

═══ LOGGING (for slow query analysis) ════════════════════════
log_min_duration_statement = 1000  # log queries slower than 1 second
log_checkpoints = on
log_connections = on
log_lock_waits = on
log_temp_files = 0  # log any query that spills to disk

═══ AUTOVACUUM ════════════════════════════════════════════════
# More aggressive autovacuum for busy tables
autovacuum_max_workers = 5
autovacuum_vacuum_cost_delay = 2ms  # default 20ms — too slow for big tables
💡 Use PGTune to generate settings
Visit pgtune.leopard.in.ua — enter your server specs (RAM, CPU, storage type, DB purpose) and it generates optimal postgresql.conf settings for you. Use this as your starting point, then measure and adjust.

Lesson 05
Monitoring & pg_stat Views
Know what your database is doing right now — and why it's slow
👨‍🏫
PostgreSQL exposes rich internal statistics through system views. These are normal SQL queries that reveal everything: which queries are running right now, which tables are getting hit most, which indexes are never used, where locks are blocking, and how much data is being read from disk vs cache. Knowing these views separates seniors from juniors.

Most important monitoring queries

sql — Active queries and blocking locks
-- What queries are running RIGHT NOW?
SELECT
    pid,
    now() - query_start           AS duration,
    state,
    wait_event_type,
    wait_event,
    LEFT(query, 80)               AS query_preview,
    client_addr,
    application_name
FROM   pg_stat_activity
WHERE  state != 'idle'
ORDER BY duration DESC;

-- Kill a runaway query (use with caution!)
SELECT pg_cancel_backend(12345);   -- graceful cancel (sends SIGINT)
SELECT pg_terminate_backend(12345); -- hard kill  (sends SIGTERM)

-- Who is blocking whom? (lock dependency tree)
SELECT
    blocked.pid                        AS blocked_pid,
    blocked_activity.query             AS blocked_query,
    blocking.pid                       AS blocking_pid,
    blocking_activity.query            AS blocking_query,
    now() - blocked_activity.query_start AS blocked_duration
FROM  pg_locks         blocked
JOIN  pg_stat_activity blocked_activity  ON blocked_activity.pid  = blocked.pid
JOIN  pg_locks         blocking          ON blocking.transactionid = blocked.transactionid
                                           AND blocking.pid != blocked.pid
JOIN  pg_stat_activity blocking_activity ON blocking_activity.pid = blocking.pid
WHERE NOT blocked.granted;
sql — Table and index statistics
-- Which tables are largest?
SELECT
    schemaname,
    relname                                       AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    pg_size_pretty(pg_relation_size(relid))       AS table_size,
    pg_size_pretty(pg_indexes_size(relid))        AS indexes_size
FROM   pg_stat_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT  20;

-- Which tables have the most sequential scans? (Missing indexes!)
SELECT
    relname          AS table_name,
    seq_scan,
    idx_scan,
    seq_scan - idx_scan          AS seq_over_idx,
    n_live_tup                   AS live_rows
FROM   pg_stat_user_tables
WHERE  seq_scan > idx_scan
  AND  n_live_tup > 10000
ORDER BY seq_scan DESC;

-- Which indexes are NEVER used? (Safe to drop = saves space + write overhead)
SELECT
    schemaname,
    relname      AS table_name,
    indexrelname AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan     AS times_used
FROM   pg_stat_user_indexes
JOIN   pg_index USING(indexrelid)
WHERE  idx_scan = 0
  AND  indisunique = FALSE    -- keep unique indexes (they enforce constraints)
ORDER BY pg_relation_size(indexrelid) DESC;
sql — Cache hit rate & slow queries
-- Cache hit ratio — should be > 99% on a healthy system
SELECT
    SUM(heap_blks_read)  AS disk_reads,
    SUM(heap_blks_hit)   AS cache_hits,
    ROUND(
        100.0 * SUM(heap_blks_hit) /
        NULLIF(SUM(heap_blks_hit) + SUM(heap_blks_read), 0),
    2) AS cache_hit_rate_pct
FROM pg_statio_user_tables;

-- Slowest queries (requires pg_stat_statements extension)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

SELECT
    ROUND(mean_exec_time::numeric, 2) AS avg_ms,
    calls,
    ROUND(total_exec_time::numeric, 2) AS total_ms,
    rows,
    LEFT(query, 80) AS query
FROM   pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT  20;

-- VACUUM & bloat status
SELECT
    relname,
    n_dead_tup                         AS dead_rows,
    n_live_tup                         AS live_rows,
    last_autovacuum,
    last_autoanalyze
FROM   pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT  10;

-- Manually trigger VACUUM on a bloated table
VACUUM ANALYZE orders;

Lesson 06
Connection Pooling
Why you can't have 10,000 direct connections — and how PgBouncer fixes it
👨‍🏫
Every PostgreSQL connection is an OS process using 5–10MB of RAM. With 1,000 app servers each keeping 10 connections, that's 10,000 connections and 100GB of RAM just for connection overhead — before a single query runs. Connection pooling solves this: a pool sits between your app and Postgres, multiplexing many app connections onto a small number of real DB connections.
App Server 1 ──┐ App Server 2 ──┤ 100 connections App Server 3 ──┼────────────────▶ ┌──────────────┐ 10 connections App Server 4 ──┤ │ PgBouncer │ ──────────────────▶ PostgreSQL App Server 5 ──┘ └──────────────┘ (5432) 100 app connections → pooled → 10 real DB connections PostgreSQL only sees 10 connections. App sees 100 available "connections".

PgBouncer pooling modes

ModeHow it worksBest for
SessionClient holds a server connection for its entire session. Least efficient.Apps that use SET, prepared statements, temp tables
TransactionClient gets a server connection only for one transaction. Returns it after COMMIT/ROLLBACK. Most popular.Most web apps — maximum connection reuse
StatementConnection returned after every single statement.Rarely used — breaks multi-statement transactions
conf — pgbouncer.ini configuration
# /etc/pgbouncer/pgbouncer.ini

[databases]
myapp = host=127.0.0.1 port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432          # PgBouncer listens here
listen_addr = 0.0.0.0
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt

pool_mode = transaction     # recommended for most apps

max_client_conn = 1000      # max connections FROM apps
default_pool_size = 20      # max connections TO postgres per DB/user
min_pool_size = 5           # keep at least 5 warm connections
reserve_pool_size = 5       # emergency extra connections
reserve_pool_timeout = 3    # wait 3s before using reserve pool

server_idle_timeout = 600   # drop idle server connections after 10min
client_idle_timeout = 0     # 0 = never drop idle clients

log_connections = 0
log_disconnections = 0
log_pooler_errors = 1

# userlist.txt format:
"appuser" "password_here"
sql — Monitor PgBouncer from inside Postgres
-- Connect to PgBouncer's admin DB
psql -h 127.0.0.1 -p 6432 -U pgbouncer pgbouncer

-- Show pool status
SHOW POOLS;

-- Show all connections
SHOW CLIENTS;
SHOW SERVERS;

-- Show stats (requests/sec, avg query time)
SHOW STATS;

-- Reload config without restart
RELOAD;
⚠️ Transaction mode limitations
In transaction pool mode: no SET session variables that persist across transactions, no advisory locks, no LISTEN/NOTIFY, no named prepared statements (some drivers handle this automatically). Check your ORM's PgBouncer compatibility docs.
Lesson 07
Security Hardening
Locking down your database for production
👨‍🏫
The default PostgreSQL install is already fairly secure — but production needs more. The principle of least privilege: your app should only have the permissions it absolutely needs. If the app user gets compromised, the damage should be limited to the data that user can see.
sql — Production security setup
═══ USER MANAGEMENT ═══════════════════════════════════════════

-- Create application user with minimal permissions
CREATE USER app_user WITH
    PASSWORD          'use-a-strong-random-password'
    CONNECTION LIMIT  50     -- hard limit on connections
    NOSUPERUSER
    NOCREATEDB
    NOCREATEROLE;

-- Grant only what the app needs
GRANT CONNECT ON DATABASE myapp TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_user;

-- Apply to future tables too
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT USAGE, SELECT ON SEQUENCES TO app_user;

-- Separate read-only user for analytics/reporting
CREATE USER readonly_user WITH PASSWORD 'another-strong-password';
GRANT CONNECT ON DATABASE myapp TO readonly_user;
GRANT USAGE ON SCHEMA public TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;

═══ ROW-LEVEL SECURITY (RLS) ══════════════════════════════════

-- Users can only see their own orders
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY orders_user_isolation ON orders
    FOR ALL
    TO app_user
    USING (user_id = current_setting('app.current_user_id')::BIGINT);

-- Your app sets this at the start of each request:
SET app.current_user_id = '123';
-- Now all queries on orders automatically filter to user 123
conf — pg_hba.conf: who can connect and how
# File: /etc/postgresql/16/main/pg_hba.conf
# TYPE  DATABASE    USER        ADDRESS           METHOD

# Allow local unix socket connections for postgres superuser only
local   all         postgres                      peer

# Allow app user from app server subnet via SCRAM (strongest auth)
host    myapp       app_user    10.0.1.0/24       scram-sha-256

# Allow readonly user from analytics subnet
host    myapp       readonly_user 10.0.2.0/24     scram-sha-256

# Deny everything else
host    all         all         0.0.0.0/0         reject

# Require SSL for all remote connections (add sslmode=require in app)
hostssl all         all         0.0.0.0/0         scram-sha-256
🚨 Security checklist for production
  • Change the postgres superuser password immediately after install
  • Never expose port 5432 to the internet — use a private network or VPN
  • Use scram-sha-256 authentication, never md5 or trust
  • Enable SSL — set ssl = on in postgresql.conf
  • Rotate passwords regularly and use a secrets manager (AWS Secrets Manager, Vault)
  • Enable log_connections and log_disconnections for audit
  • Use Row Level Security for multi-tenant applications
Lesson 08
Must-Know Extensions
PostgreSQL's superpower: a rich extension ecosystem
👨‍🏫
PostgreSQL can be extended with C plugins that become first-class database features. Extensions add new data types, index methods, functions, and even entire subsystems. The extension ecosystem is one of the biggest advantages PostgreSQL has over every other database.
sql — Installing and using extensions
-- Install an extension (requires superuser)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- See all installed extensions
SELECT name, default_version, installed_version, comment
FROM pg_available_extensions
WHERE installed_version IS NOT NULL
ORDER BY name;

sql — pg_trgm: fast fuzzy search with LIKE '%text%'
CREATE EXTENSION pg_trgm;

-- Create a GIN trigram index
CREATE INDEX idx_products_name_trgm
    ON products USING GIN(name gin_trgm_ops);

-- Now LIKE '%wireless%' uses the index — previously it would seq scan
SELECT * FROM products
WHERE name ILIKE '%wireless%';

-- Similarity score: 0.0 (no match) to 1.0 (exact match)
SELECT name, similarity(name, 'wireles headphone') AS score
FROM products
WHERE name % 'wireles headphone'   -- % = similarity > threshold
ORDER BY score DESC;
sql — pgvector: store AI embeddings
CREATE EXTENSION vector;

-- Store 1536-dimensional OpenAI embeddings
CREATE TABLE documents (
    id        BIGSERIAL PRIMARY KEY,
    content   TEXT,
    embedding vector(1536)  -- OpenAI text-embedding-ada-002
);

-- Index for fast nearest-neighbour search
CREATE INDEX ON documents
    USING ivfflat(embedding vector_cosine_ops) WITH (lists = 100);

-- Find the 5 most semantically similar documents to a query embedding
SELECT id, content,
    embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM documents
ORDER BY distance
LIMIT 5;
sql — pg_cron: schedule jobs inside PostgreSQL
CREATE EXTENSION pg_cron;

-- Refresh materialized view every 15 minutes
SELECT cron.schedule(
    'refresh-daily-revenue',      -- job name
    '*/15 * * * *',               -- every 15 minutes
    'REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue'
);

-- Delete old audit logs every Sunday at 3am
SELECT cron.schedule(
    'cleanup-audit-log',
    '0 3 * * 0',
    'DELETE FROM audit_log WHERE changed_at < NOW() - INTERVAL ''90 days'''
);

-- List scheduled jobs
SELECT * FROM cron.job;

-- Remove a job
SELECT cron.unschedule('refresh-daily-revenue');
